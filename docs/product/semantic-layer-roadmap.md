# 发票业财系统语义层先行路线图

版本：V1.0  
日期：2026-07-09  
适用范围：发票中心、客户结算、供应商结算、代垫款管理、应收与回款、税务合规、报表中心、基础配置  
目标形态：在设计初期建立统一业务语义层，为后续原型深化、真实后端迁移、链票对接、报表分析和智能化能力提供长期底座。

## 1. 建设目标

语义层先行的目标不是多做一层配置，而是让发票业财系统从一开始就拥有统一业务语言。

同一个客户、业务编号、发票金额、代垫余额、逾期金额、红冲状态，在页面、流程、接口、报表、后端数据模型里都应指向同一套定义。

最终系统不应靠一个个页面堆出来，而应由以下内容驱动：

- 业务对象
- 字段口径
- 状态流程
- 业务动作
- 对象关系
- 指标口径
- 页面视图

这条路径最终会让系统更容易继续扩展到真实业财一体系统。后续接链票、接数据库、做报表、做权限、做审计、做 AI 问数，都可以围绕同一套语义底座展开。

## 2. 分阶段路线

### Phase 0：现有原型语义盘点

目标：先把当前系统里已经出现的业务语言整理出来。

主要工作：

- 梳理业务域地图：发票中心、客户结算、供应商结算、代垫款、应收与回款、税务合规、报表中心、基础配置。
- 梳理核心业务对象：开票申请、销项发票、进项发票、结算单、成本单、代垫单、应收单、银行流水、链票流水。
- 梳理字段字典初稿：业务编号、客户、供应商、报关单号、运单号、仓单号、发票号码、金额、税额、状态、账龄等。
- 梳理状态词统一问题：例如待审核与待审批、已核销与已销账、处理中与同步中等。
- 标记字段来源：来自业务单据、发票、链票、财务、税务、银行流水、人工录入或系统计算。

阶段效果：

- 可以看到系统的完整业务地图。
- 现有页面不再只是页面清单，而能归属到具体业务对象和业务链路。
- 后续新增字段、页面、流程时，有基础口径可以复用。

### Phase 1：建立业务语义字典

目标：定义系统的标准业务语言。

主要工作：

- 定义标准业务对象：每个对象有唯一编码、中文名、业务说明、主键字段、所属业务域。
- 定义标准字段：每个字段有中文名、字段类型、业务含义、是否可筛选、是否可排序、是否金额字段、是否敏感字段。
- 定义标准状态：每个业务对象的状态集合和状态流转。
- 定义标准动作：提交审核、同步链票、申请红冲、作废校验、收款认领、核销、归档等动作归属到具体对象。
- 定义标准指标：待开票金额、未收应收、代垫余额、逾期金额、坏账风险、进项可抵扣税额、项目毛利率。

阶段效果：

- 页面字段、筛选条件、报表指标不再各写各的。
- 后续新增页面时，优先复用语义字典，而不是重新命名字段。
- 产品、财务、税务、业务和开发可以围绕同一套口径沟通。

### Phase 2：语义视图驱动核心页面

目标：让语义层开始影响真实页面，但不一次性重构全部页面。

优先试点页面：

- 开票申请
- 销项发票
- 代垫登记
- 应收台账

主要工作：

- 为每个页面建立语义视图。
- 由语义视图定义默认列、更多搜索条件、详情抽屉字段、批量动作、主按钮。
- 列设置、字段排序、更多筛选、详情展示开始从语义视图生成。
- 保留现有 UI 和 mock 数据，降低返工风险。

阶段效果：

- 页面仍然保持当前可演示效果，但背后开始有统一业务模型。
- 新增字段时，先进入语义字典，再决定哪些页面展示。
- 列表、筛选、详情、批量操作的规则更稳定。

### Phase 3：语义层驱动业务流程

目标：从字段统一升级到流程统一。

重点覆盖流程：

- 结算单生成开票申请
- 开票申请提交链票
- 链票返回销项发票
- 发票生成应收
- 银行流水认领
- 应收核销
- 代垫确认、开票、回款、核销
- 红冲、作废、归档

主要工作：

- 为每个业务对象定义状态机。
- 为每个业务动作定义前置条件、结果状态、关联对象变化。
- 将链票对接动作抽象为标准接口事件，例如蓝字开票、红冲申请、状态同步、版式下载、失败重试。
- 将异常、风险、审批、日志挂接到统一业务动作和业务对象。

阶段效果：

- 系统能清楚表达为什么某个按钮能点或不能点。
- 详情页能展示完整业务追溯。
- 后续接真实后端时，接口和流程不容易推倒重来。

### Phase 4：语义层对接数据模型和接口

目标：为真实系统建设做迁移准备。

主要工作：

- 语义对象映射数据库表。
- 语义字段映射 API DTO。
- 语义指标映射报表查询。
- 链票接口字段映射到内部开票、红冲、归档对象。
- 操作日志、接口日志、审计日志引用同一套对象编号和动作编号。

阶段效果：

- 原型可以平滑迁移成真实系统。
- 前端页面、后端接口、数据库表、报表口径有一致来源。
- 后续增加权限、审批、审计、数据权限时有明确抓手。

### Phase 5：语义层成为系统能力中心

目标：让语义层不仅服务页面，还服务经营分析和智能化。

最终能力：

- 管理层看报表时，指标口径来自语义层。
- 财务配置规则时，字段和状态来自语义层。
- 业务追溯时，系统知道对象之间的关系。
- AI 问数时，可以理解“这个客户代垫还有多少没收”、“本月红冲影响多少收入”、“哪些报关单关联的发票未归档”等问题。
- 新增业务模块时，先建语义对象，再生成页面、流程、报表和接口契约。

## 3. 最终实现效果

最终系统会形成四层结构。

### 3.1 语义层

定义业务对象、字段、状态、动作、关系、指标。

示例：

- 业务对象：开票申请、销项发票、进项发票、代垫单、应收单、链票流水。
- 字段：业务编号、客户名称、发票号码、含税金额、税额、报关单号、运单号、仓单号。
- 状态：待审核、待链票开具、已开票、已交付、红冲申请中、已归档。
- 动作：提交审核、同步链票、申请红冲、作废校验、收款认领、核销。
- 指标：待开票金额、未收应收、代垫余额、逾期金额、项目毛利率。

### 3.2 业务流程层

处理开票、收票、红冲、作废、代垫、应收、核销、税务等流程。

语义层会告诉流程层：

- 哪些状态下可以执行某个动作。
- 执行动作后对象状态如何变化。
- 哪些关联对象需要同步更新。
- 哪些异常需要进入风险或待办。

### 3.3 页面体验层

列表、筛选、详情、按钮、报表从语义层派生，保持 B 端一致性。

页面层不再自己随意定义字段和动作，而是从语义视图中读取：

- 默认展示列
- 更多搜索条件
- 可排序字段
- 批量操作
- 详情抽屉字段
- 主操作按钮
- 风险提示和操作日志

### 3.4 数据与接口层

数据库、链票接口、报表查询、日志审计都映射回统一语义。

后续真实系统建设时，可以将语义对象映射为数据库表和接口契约，将语义指标映射为报表查询口径，将语义动作映射为审计日志和流程事件。

## 4. 路径价值

这条路径的核心价值是降低后续返工。

短期价值：

- 让当前原型更有系统设计感。
- 让页面字段、筛选、详情、按钮更统一。
- 让产品文档、原型页面和代码表达一致。

中期价值：

- 更容易迁移到真实后端。
- 更容易接链票接口。
- 更容易扩展报表中心。
- 更容易维护复杂业务流程。

长期价值：

- 支撑业财一体系统持续扩展。
- 支撑权限、审计、规则配置和数据治理。
- 支撑 AI 问数和智能助手理解业务。
- 支撑跨模块追溯，例如从发票追到业务、结算、代垫、应收、回款、链票流水。

## 5. 验收信号

当以下现象出现时，说明语义层路线开始产生价值：

- 同一个字段在不同页面不再出现多套叫法。
- 每个主要页面都能说明自己基于哪个业务对象。
- 每个按钮都能说明前置条件和业务结果。
- 每个核心指标都能追溯计算口径。
- 任意一张发票都能追溯到业务编号、客户、结算、代垫、应收、回款、链票流水。
- 后续接数据库和链票真实接口时，不需要重新设计页面字段和业务对象。

## 6. 当前默认假设

- 第一阶段仍然是原型系统，不做真实数据库、登录权限、真实链票调用。
- 语义层先服务业务系统建设，不先做重型 BI 平台。
- 采用渐进路线，先接管核心页面，再逐步扩展到全部模块。
- 最终目标是为真实业财一体系统沉淀长期可复用的业务语言和系统骨架。
