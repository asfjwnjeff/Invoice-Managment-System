# 半导体供应链发票业财一体系统 PRD

版本：V1.1
日期：2026-07-09
适用范围：发票中心、客户结算、供应商结算、代垫款管理、应收与回款、税务合规、链票对接、报表中心、基础配置  
目标形态：先建设可演示原型，后续迁移为真实业财一体系统中的发票与票财税核心模块。

## 1. 背景与问题

公司为半导体行业供应链服务企业，业务覆盖进口关务、国内外运输、保税/非保税仓储、客户代垫、供应商成本、收入结算、开票、回款和税务管理等全链路。当前开票及税务侧能力使用链票，财务系统和发票管理依赖外采小系统，业务、发票、收入、成本、代垫、应收、税务之间存在数据割裂。

现状痛点：

- 开票依赖链票，但开票前的业务申请、审核、抬头校验、商品编码、收入/应收联动缺少统一承载。
- 进项发票、成本确认、供应商对账、暂估应付、预付款核销和成本分摊之间缺少清晰闭环。
- 代垫场景复杂，尤其关税、进口增值税、保证金、查验费等与客户确认、开票、应收、回款核销的关联不足。
- 半导体业务需要按业务编号、报关单号、运单号、仓单号、项目、客户、供应商进行全链路追溯。
- 红冲、作废、交付、归档、查验、认证抵扣、风险巡检、链票接口日志等能力分散。
- 管理层需要按客户、业务线、项目、区域、税种、账龄、毛利、代垫超期等维度分析经营质量。

## 2. 产品目标

建设一套集成在自建业财一体系统中的发票业财中台，连接业务单据、链票开票、客户结算、供应商结算、代垫、应收与回款、税务合规和报表中心。

核心目标：

- 打通从业务委托到开票、收票、入账、应收、回款、成本、代垫、归档的全链路。
- 对接链票完成实际开票、红冲、票源、版式文件、状态同步和接口回调。
- 支持半导体供应链业务中的关务、运输、仓储、代垫、供应商成本和客户对账。
- 降低重复开票、错误税率、抬头错误、代垫超期、成本漏入账、应收逾期等风险。
- 为财务、税务、业务、运营和管理层提供统一数据底座和报表视角。

## 5. 业务主流程

### 5.1 销项开票流程

1. 业务系统产生结算单或收入确认单。
2. 系统生成开票申请。
3. 校验客户抬头、税号、商品/服务编码、税率、开票主体、合同余额、信用额度。
4. 进入开票审核。
5. 审核通过后提交链票开具。
6. 链票返回发票号码、状态、版式文件。
7. 系统生成销项发票记录并联动应收台账。
8. 发票交付客户并记录回执。
9. 发票归档，进入报表中心。

### 5.2 进项收票与成本流程

1. 成本单来自供应商对账、运输单、仓储单、报关服务单或采购合同。
2. 发票通过上传、邮箱、链票同步、税局下载或 OCR 进入进项采集。
3. 系统查验真伪、查重、校验抬头税号。
4. 进项发票与成本单、供应商对账单、付款单进行三单/四单匹配。
5. 可抵扣发票进入认证抵扣。
6. 发票入账，冲回暂估或产生差异调整。
7. 形成成本分析、应付分析、供应商成本报表。

### 5.3 关务代垫流程

1. 关务业务产生报关单、税单、查验单或保证金单。
2. 系统登记代垫单，记录客户、报关单号、HS 编码、完税价格、关税、进口增值税、币种、汇率。
3. 财务付款形成代垫余额。
4. 发送客户确认。
5. 客户确认后按规则生成代垫开票申请或代垫应收。
6. 回款进入银行流水。
7. 系统进行收款认领和代垫核销。
8. 超期未回款进入催收、减免或坏账流程。

### 5.4 应收与回款流程

1. 销项发票、代垫应收、未开票收入生成应收台账。
2. 银行流水导入或银企直连同步。
3. 系统按客户、金额、备注、发票号、业务编号推荐认领。
4. 自动或手工核销应收。
5. 逾期应收进入催收计划。
6. 长账龄或高风险客户进入坏账计提、核销、转回。

## 10. 关键规则

### 10.1 开票规则

- 同一业务单可生成多张发票，但必须保留业务单与发票关系。
- 开票前必须校验客户抬头、税号、商品编码、税率、开票主体。
- 半导体关务服务、运输服务、仓储服务可配置不同商品/服务编码和税率。
- 客户信用额度不足时，开票申请进入异常挂起或高级审批。

### 10.2 红冲作废规则

- 数电票已开具后原则上不走传统作废，应引导红冲。
- 支持全额红冲、部分红冲、红冲后重开。
- 红冲必须保留原票关系、红票关系和链票流水。

### 10.3 代垫规则

- 关税、进口增值税、保证金、查验费必须关联客户、业务编号和报关单号。
- 客户确认前不得自动生成代垫开票申请。
- 超过账期未回款的代垫进入催收计划。
- 可按费用类型配置是否开票、是否计入应收、是否允许减免。

### 10.4 成本规则

- 成本可先确认，发票未到时进入暂估应付。
- 发票到票后自动匹配成本单并冲回暂估。
- 成本差异需要走调整审批。
- 供应商对账确认后才能进入应付或付款流程。

### 10.5 核销规则

- 银行流水可按客户、金额、备注、发票号、业务编号自动推荐匹配。
- 支持部分核销、批量核销、尾差处理和反核销。
- 坏账核销必须保留应收追溯链。

## 17. 语义层建设路线

### 17.1 建设目标

系统应在设计初期建立统一业务语义层，让同一个客户、业务编号、发票金额、代垫余额、逾期金额、红冲状态，在页面、流程、接口、报表、后端数据模型中指向同一套定义。

语义层应逐步沉淀：

- 业务对象
- 字段口径
- 状态流程
- 业务动作
- 对象关系
- 指标口径
- 页面视图

### 17.2 分阶段路线

Phase 0：现有原型语义盘点。

- 梳理业务域地图。
- 梳理核心业务对象。
- 梳理字段字典初稿。
- 统一状态词和字段来源。

Phase 1：建立业务语义字典。

- 定义标准业务对象。
- 定义标准字段。
- 定义标准状态。
- 定义标准动作。
- 定义标准指标。

Phase 2：语义视图驱动核心页面。

- 优先试点开票申请、销项发票、代垫登记、应收台账。
- 由语义视图定义默认列、更多搜索条件、详情抽屉字段、批量动作和主按钮。
- 保留现有 UI 和 mock 数据，降低返工风险。

Phase 3：语义层驱动业务流程。

- 定义业务对象状态机。
- 定义业务动作前置条件和结果状态。
- 将链票动作抽象为标准接口事件。
- 将异常、风险、审批、日志挂接到统一业务对象和业务动作。

Phase 4：语义层对接数据模型和接口。

- 语义对象映射数据库表。
- 语义字段映射 API DTO。
- 语义指标映射报表查询。
- 链票接口字段映射内部对象。
- 操作日志、接口日志、审计日志引用统一对象编号和动作编号。

Phase 5：语义层成为系统能力中心。

- 报表指标口径来自语义层。
- 财务规则配置引用语义字段和状态。
- 业务追溯基于语义对象关系。
- AI 问数可以理解发票、代垫、应收、回款、链票等业务问题。

### 17.3 最终四层结构

1. 语义层
   定义业务对象、字段、状态、动作、关系和指标。

2. 业务流程层
   处理开票、收票、红冲、作废、代垫、应收、核销和税务流程。

3. 页面体验层
   列表、筛选、详情、按钮和报表从语义层派生，保持 B 端一致性。

4. 数据与接口层
   数据库、链票接口、报表查询和日志审计映射回统一语义。

### 17.4 验收信号

- 同一个字段在不同页面不再出现多套叫法。
- 每个主要页面都能说明自己基于哪个业务对象。
- 每个按钮都能说明前置条件和业务结果。
- 每个核心指标都能追溯计算口径。
- 任意一张发票都能追溯到业务编号、客户、结算、代垫、应收、回款和链票流水。
- 后续接数据库和链票真实接口时，不需要重新设计页面字段和业务对象。
