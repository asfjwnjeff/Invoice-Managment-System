# 发票业财系统调研与范围基线

更新时间：2026-07-06
本文档用于沉淀前期调研、你们公司的现状信息、当前原型状态，以及后续调整一级菜单、二级菜单、页面字段和交互时的参考基线。后续实现时，优先以本文档作为需求校验依据。

## 1. 项目背景
你们现在的系统格局：
- 开票及税务对接：当前使用链票，后续仍希望通过链票完成开票、红冲、票源、状态回写等税务侧能力。
- 财务系统及发票管理：当前使用外采的小系统，能力相对割裂。
- 新系统方向：公司正在自建业财一体系统，希望把发票管理作为其中的核心模块，而不是继续依赖外部小系统。
- 目标形态：自建一套完整的发票系统，集成在业财一体系统中，对接链票完成开票全流程，同时覆盖收入、成本、代垫、应收回款、税务分析、档案和风险管理。
当前原型目标：
- 做成完整可演示的原型系统。
- 暂不需要真实接口、真实数据库持久化、账号登录和权限控制。
- 页面交互、业务流程、模块完整度和 B 端商务风格要尽量接近成熟产品。
- 技术栈目前采用 Next.js，方便本地长期运行，也方便后续迁移和 Render 演示部署。
- Render 部署相关说明不放到页面内容里，只保留在项目文档和部署配置里。

## 2. 你们现有页面与业务线索
从你们已有系统页面中，已经参考过这些业务模块和页面习惯：
- 收入相关：开票申请、收入发票查询、未开票收入、已开票收入。
- 发票相关：发票池、进项发票、销项发票、红冲、作废、交付、链票流水、归档、风险。
- 成本相关：成本发票管理、预付款关联发票、应付匹配。
- 代垫相关：代垫登记、代垫查询、代垫发票查询、区域超期汇总、客户超期汇总。
- 应收销账相关：应收台账、银行流水维护、银行流水销账、逾期管理、坏账管理。
- 报表相关：项目毛利、发票分析、税务分析、代垫分析。
- 设置相关：企业主体/抬头、客户档案、供应商档案、税务规则、业务字段、业务票池、导出模板、系统日志。
现有系统页面给原型带来的设计约束：
- 页面应保持 B 端商务系统风格，强调效率、稳定、可扫描，不做营销式页面。
- 按钮要是业务动作，例如提交审核、同步状态、重试、导出、匹配、核销、归档，不要出现“平台内交付、邮件交付、推送第三方”这类像渠道入口但缺少业务语义的按钮。
- “更多搜索条件”必须真实可交互。如果页面没有更多条件，就隐藏该入口，避免出现禁用但无法解释的按钮。
- 搜索区、表格、详情抽屉、流程状态、风险提示和操作日志应成为统一页面骨架。

## 3. 市场成熟系统能力参考
公开调研参考的成熟产品和能力方向包括：
- 百望云：数电票、进项管理、红冲、发票采集、税务合规和接口能力。
- 金蝶发票云：开票、收票、查验、归档、发票与财务业务联动。
- 泛微业票通：发票采集、验真、报销入账、档案和业务协同。
- 用友、金蝶等业财系统常见能力：收入、应收、成本、应付、对账、核销、凭证、税务分析的业务闭环。
成熟系统通常不是只有“发票列表”，而是覆盖以下闭环：
1. 业务单据生成开票或收票需求。
2. 发票申请、审核、开具、红冲、作废、交付。
3. 进项票采集、查验、查重、认证抵扣、入账。
4. 发票与收入、成本、应收、应付、代垫、回款、付款联动。
5. 发票档案、版式文件、原始文件、验签、借阅和归档。
6. 税务规则、商品编码、税率、额度、票源和接口状态管理。
7. 异常、风险、逾期、坏账、差异、重试和审计日志闭环。

## 4. 当前原型现状
当前原型已有一级菜单：
- 工作台
- 收入管理
- 发票管理
- 成本管理
- 代垫管理
- 应收销账
- 报表分析
- 系统设置
当前主要不足：
- 成本管理只有成本发票、预付款关联、应付匹配，缺少成本确认、供应商对账、暂估、分摊、调整等成熟系统能力。
- 发票管理已有主干，但缺少发票采集中心、查验查重、认证抵扣、发票入账、开票审核等关键环节。
- 收入管理缺少收入确认、客户对账、收入调整、收入发票匹配等业财闭环。
- 代垫管理缺少客户确认、代垫开票、代垫回款核销、减免/坏账等闭环。
- 应收模块已有逾期和坏账，但缺少收款认领、催收计划、自动核销规则、客户信用额度等成熟能力。
- 税务能力分散在发票和设置中，建议独立成税务管理视角。
- 报表分析偏少，需要补足开票、收票、抵扣、异常、应收账龄、应付账龄、暂估余额等分析。
- 设置模块缺少商品服务编码、开票规则、审批流程、单据编号、匹配规则、核销规则、风险规则、链票接口配置等基础能力。

## 5. 目标一级菜单

后续建议调整为 B 版成熟业务信息架构：

1. 经营工作台
2. 发票中心
3. 客户结算
4. 供应商结算
5. 代垫款管理
6. 应收与回款
7. 税务合规
8. 报表中心
9. 基础配置

说明：

- “收入管理”调整为“客户结算”，强调客户对账、未开票收入、已开票收入和收入发票匹配。
- “成本管理”调整为“供应商结算”，覆盖成本单、供应商对账、成本发票匹配、应付匹配、预付款和暂估。
- “应收销账”调整为“应收与回款”，范围更完整，覆盖应收、银行流水、认领、核销、逾期、催收、坏账和客户信用。
- “税务管理”调整为“税务合规”，把税额、抵扣、税负、票源、额度、申报状态和税务规则集中表达。
- “系统设置”调整为“基础配置”，更符合 B 端业务系统习惯。

## 8. 链票对接边界
自建系统和链票的建议分工：
- 自建系统负责业务流程、申请审核、状态编排、数据归集、收入成本代垫应收联动、风险提示、报表分析和用户交互。
- 链票负责实际开票、红冲、票源、税控设备、版式文件、税务状态、链票流水和税务侧接口能力。
- 自建系统需要保存链票业务流水号、请求参数摘要、返回状态、错误码、失败原因、重试次数、回调时间、版式文件链接和最终票据状态。
需要在原型中体现的链票动作：
- 提交开票
- 查询开票结果
- 下载版式文件
- 发票交付
- 发起红冲
- 查询红冲结果
- 查询/同步发票状态
- 查询票源或额度
- 失败重试
- 查看接口日志
特别注意：
- 数电票规则与传统纸票/电票不同，已开具后的作废能力需要谨慎表达，更多场景应引导到红冲。
- 原型中可以保留“作废管理”，但页面文案和规则要区分适用票种，避免误导。

## 10. 后续实现原则
菜单和页面调整时遵循以下原则：
- 先补齐信息架构，再补页面字段和交互，最后再考虑真实接口。
- 页面要体现成熟系统能力，但不要把所有功能都做成复杂流程。原型阶段重点是可演示、可理解、可迁移。
- 一级菜单负责业务域，二级菜单负责业务对象或流程环节，避免把按钮动作做成菜单。
- 操作按钮保持商务系统语义，如提交审核、确认、匹配、核销、重试、同步、归档、导出。
- 搜索条件应按成熟系统常见字段配置，例如日期、状态、主体、客户、供应商、票种、税率、业务编号、发票号码、链票流水、异常类型。
- 表格列应优先展示单号、对象、主体、金额、税额、状态、来源、所属期间、经办人、更新时间。
- 详情抽屉要展示基础信息、业务关联、发票信息、金额税额、流程记录、链票流水、风险提示和操作日志。
- 报表页面应支持下钻到明细，避免只展示静态指标。
- 后续如果迁移到真实后端，应保留当前页面结构，把 mock 数据替换为接口数据。
